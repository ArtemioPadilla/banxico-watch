# Site & codebase audit — June 2026

A full review of the **deployed site** (https://artemiop.com/inceptor/, commit
`f9a95b8` — deploy confirmed current) and the **codebase**, covering design,
UX, UI, functionality, accessibility, SEO, performance, tests, and CI. Live
behavior was verified in Chrome (desktop, dark + light themes); code findings
come from a file-by-file review of `src/components/ui/`, islands, stores,
`src/lib/`, pages, layouts, content collections, styles, tests, workflows, and
both server archetypes.

Severity scale: **P0** = broken in production or data-loss/security risk;
**P1** = visibly wrong, misleading, or a real a11y/SEO failure; **P2** =
quality, consistency, DX. Every code finding carries `file:line`.

---

## 1. Executive summary

The site looks polished at first glance — strong editorial typography, a
coherent emerald identity, live-rendered components — but **three of its four
flagship interactive demos misbehave in production**, the docs search appears
dead, several "marketing" numbers are wrong, and the site has no footer, no
custom 404, and no SEO head plumbing (canonical/hreflang/robots sitemap). The
component library is genuinely good; the test suite is wide but shallow
(source-text assertions dominate); CI double-builds, never runs the backend
test suites, and isn't protected against the PR-burst event drops we hit this
very week.

## 3. P0 — broken in production

### 3.1 `DataCloneError` from the TanStack Query persister *(Top 10 #1)*

Reproduced on `/demos/dashboard/` and again on a second page load — it fires
on every page that mounts `QueryProvider` while a query is in flight:

```
DataCloneError: Failed to execute 'put' on 'IDBObjectStore':
#<Promise> could not be cloned.   (QueryProvider.Bo8Ojfp_.js)
```

Root cause, `src/lib/queryClient.ts:17`: `persistClient: async (client) =>
set(idbKey, client)` writes the dehydrated client straight into IndexedDB.
TanStack Query v5's `dehydrate()` **includes pending queries by default**, and
a dehydrated pending query carries its `promise` — which structured clone
rejects. The `shouldDehydrateQuery: (q) => Boolean(q.meta?.persist)` filter
checks **only** the meta flag, not query status — a pending query with
`meta.persist` sails straight into IndexedDB.

**Fix (either):**
- Exclude in-flight queries:
  `shouldDehydrateQuery: (q) => q.state.status === 'success' && Boolean(q.meta?.persist)`
  — or compose TanStack's own default:
  `defaultShouldDehydrateQuery(q) && Boolean(q.meta?.persist)`.
- And/or make the persister JSON-safe:
  `await set(idbKey, JSON.parse(JSON.stringify(client)))` (crude) or strip
  `promise` from dehydrated queries before `set`.

Side effect worth noting: the error is caught by HydrationCanary/FAB
diagnostics, so **every visitor to the dashboard sees a red error badge on the
FAB** — the ethics/trust surface reporting the template's own bug.

Related (same file): in `attachPersister` the returned `unsubscribe` from
`persistQueryClient` is used as the React effect cleanup
(`QueryProvider.tsx:22-25`). Verify it doesn't call `removeClient()` — under
StrictMode double-invoke that wipes the persisted cache on every dev mount.

### 3.2 50k-row table: "No results" until first scroll *(Top 10 #2 — P1: works after one scroll)*

On load, the table shows the header, **"No results."** in the body, and
"Showing 0 of 50000 in viewport". One scroll tick later all rows appear and it
behaves beautifully. The TanStack Virtual instance measures a 0-height scroll
container on mount (hydration happens before layout, or `client:visible` fires
before the container has its height) and never re-measures without a scroll
event.

**These are two independent fixes — file as two issues:**

1. *Virtualizer first-paint measurement:* trigger `virtualizer.measure()` in a
   `useEffect` after mount / `ResizeObserver` on the scroll container, or
   render an initial window (`overscan` + non-zero `estimateSize` fallback) so
   first paint is populated.
2. *Wrong empty-state gate:* "No results." renders while 50k rows exist —
   gate it on `rows.length === 0` (data), not on the virtual items array
   (viewport). Separate root cause, separate test.

### 3.3 Dashboard charts render empty *(Top 10 #3)*

With data present (KPIs fill in: 0 open issues / 5 PRs / 2 authors; the
bar-list shows real counts):

- **"Issues by label (top 8)"** draws axes + category ticks
  (`type:feat`, `phase-7`, …) but **zero bars**.
- **"Open vs closed"** draws only its legend — no donut segments.

Likely causes to check in `src/components/ui/charts/*` +
`DashboardIsland.tsx`: data-key mismatch between series and the bar `dataKey`;
donut receiving `open: 0` and filtering out zero-value slices entirely (0 open
is legitimate — closed should still render); the deprecated Recharts `Cell`
usage flagged by astro-check hints in `donut-chart.tsx`.

Also: KPI placeholders render as a bare `"…"` string with no skeleton; the
table inside the dashboard shows "No results." while loading (same wrong
empty-state-vs-loading conflation as 3.2). Loading ≠ empty — use `Skeleton`
(it's in the gallery!) and reserve "No results" for a successful empty
response.

### 3.4 Docs search silently dead *(Top 10 #4)*

Typing in the docs search box produced no dropdown, no network fetch, no
console error. Code review explains it: `DocsSearch.astro:40-42` builds the
bundle URL as `'/' + '_pagefind' + '/pagefind.js'` — root-absolute. Verified
live: `/_pagefind/pagefind.js` → **404**, `/inceptor/_pagefind/pagefind.js` →
**200**. Under the subpath deploy the dynamic import can never succeed.

**Fix:** `const url = import.meta.env.BASE_URL.replace(/\/$/, '') +
'/_pagefind/pagefind.js'` (`BASE_URL` is available inside Astro component
scripts). Acceptance criteria for the Playwright smoke test: visit `/docs/`,
type a query, assert ≥1 result link — and run it against **both** base
configs (`/` and `/inceptor/`), since the bug only exists under a subpath.
This was invisible to the whole `?raw` test suite.

### 4.2 Homepage

- **The copy leads with the stack, not the methodology** — "A zero-cost,
  static-first scaffold … Astro 5, React 19, Tailwind v4, shadcn/Base UI,
  TanStack, Tremor Raw…". `docs/POSITIONING.md` §1 explicitly decided the
  opposite: *lead with the methodology, stack is table stakes, mention it
  third*. The hero subcopy reads as a parts list with seven brand names; the
  loop ("How a feature ships") — the actual differentiator — is the last
  section. Restructure: hero → the loop → proof (stats) → the kit.
- **Stats strip is hardcoded and wrong** (`index.astro:11-15`): `'60'` pages,
  `'439'` tests as string literals (the build emits 68 pages, and 476 tests
  are green). Only `componentCount` is computed. Wire all four to build-time
  values — a template selling "mechanical, not aspirational" cannot ship
  hand-typed metrics. Bonus inconsistency: gallery hero says **"All 19
  components across 14 categories"** while the home stats say **"44
  components"** and the kit section says "browse all 19" (19 = gallery
  entries, 44 = ui files — pick one vocabulary: "19 gallery categories · 44
  components").
- **The page is short and ends abruptly** (no footer, §4.1): hero → stats →
  6 kit cards → loop → one-line FAB note. Missing: any link to
  `docs/PRINCIPLES`/`ETHICS` (the governance wedge), the `create-inceptor-app`
  one-liner (`npm create inceptor-app` belongs in the hero as the primary
  CTA), social proof / FinSight as reference implementation.
- Orbit animation: charming, but the three orbit chips (`issue`, `review`,
  `merge`) are nearly invisible at rest and it's unclear they relate to the
  loop section far below. Consider animating the loop steps instead.

### 4.3 Gallery

- Index renders **live components on one tall page** — great differentiator
  vs static screenshots; keep it. But per-category cards bury their "open →"
  link in the top-right corner in tiny mono text; make the whole card header
  clickable.
- `/gallery/primitives/` detail is the best page on the site (light+dark
  side-by-side, install command, source link, recipes). Gaps: **no props/API
  table**, no variant matrix promised by the index copy ("variant matrix and
  source"), and the copy-button on snippets needs verifying against #119.
- URL guessing fails hard: `/gallery/button/` → GH 404. With 19 entries, add
  category aliases or at least the custom 404 with a "did you mean" link list.
- Theme-pair preview contrast:
  - *Light mode:* LIGHT panel on a near-white page with a hairline border —
    acceptable.
  - *Dark mode:* the LIGHT panel **glows at full white** inside a dark page —
    high-contrast shock that makes the adjacent dark panel look muddy. Add a
    soft border or inset shadow on the light panel when the page is dark.

### 4.4 Demos

- Demos index: 5 cards, no thumbnails, no "what to look for" hints, and no
  links to the source files of each demo — the audience is developers;
  every demo page should link its own `.astro`/`.tsx` on GitHub
  ("View source" is the template's whole point).
- Dashboard (beyond §3.3): chart palette uses near-identical emerald/teal
  tones for adjacent series — add hue contrast; sparklines have no axis
  context (fine) but also no tooltip/aria description (not fine — see chart
  a11y in §5).
- `data.astro:22` still links the pre-redirect `/data/large` path — works
  only through the 301 (`demos/data.astro:22`); update to
  `withBase('/demos/data/large/')`.

### 4.5 Docs

- Solid IA (sidebar groups, ⌘K affordance, prev/next), but **19 of 36 pages
  are "being migrated" stubs** that just link to GitHub (all of `ethics-ux/`,
  most of `how-we-work/` and `building/`, `stack/overview`, `reference/*`,
  `history/*` — grep-verified). More than half the docs surface is hollow:
  they occupy sidebar slots and will pollute the search index the moment
  Pagefind is fixed. Either migrate the content or pull them from the sidebar
  until ready.
- Two ADRs share the number **0003** (centinela-verdict-tokens and
  starlight-superseded) in the sidebar (`docs-sidebar.ts:87-91`) — renumber or
  visually mark the superseded one.
- `what-you-get.md:10-15` still documents the old routes (`/showcase`,
  `/dashboard`, `/data`…) and calls the docs engine "(Starlight)" — both
  superseded by ADR 0004.
- No "Edit this page" link per doc — one line in `DocsLayout`, high
  trust/contribution payoff for a template whose pitch is issue-driven
  contribution.

### 4.6 Blog & i18n

- Blog ships **one placeholder post** ("Welcome to your blog — replace with
  something real, or delete the blog"). On the template's own marketing
  deploy, this *is* the demo content. Either write 2–3 real posts (changelog

- `/es/` is a **different, older site**: centered legacy hero ("Lanza más
  rápido. Lanza con principios."), no stats, no kit grid, no loop section, nav

  demo. There's also **no language switcher anywhere** on the EN side (the ES
  page has an "English" button; EN has nothing) and no `hreflang` alternates

— breaks on every fork; assert URL shape instead.
